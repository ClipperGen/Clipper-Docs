# Laporan Pemangkasan Pipeline Clipper-Gen: Pembuatan Klip Akhir ke Ujung

## Ringkasan Eksekutif
Fitur inti platform Clipper-Gen — pembuatan klip portrait otomatis dari video YouTube — rusak di beberapa titik dalam pipeline. Backend Go membuat proyek tetapi tidak pernah memicu pencernaan video. Mesin pemrosesan Python menganalisis video dan menghasilkan rekomendasi klip tetapi tidak pernah memicu rendering. Dan frontend tidak memiliki mekanisme untuk menampilkan progres pemrosesan atau mengambil klip yang sudah dirender. Sesi ini menelusuri pipeline penuh dari ujung ke ujung, mengidentifikasi setiap tautan yang rusak, dan menghubungkannya. Platform sekarang menyelesaikan alur penuh: pengajuan URL YouTube, unduh video otomatis, transkripsi, analisis AI, pembuatan kandidat klip, rendering portrait dengan subtitle burned, dan tampilan frontend dengan status polling live.

## Detail Bug & Resolusi

### 1. Pembuatan Proyek Tidak Pernah Memicu Pencernaan Video (KRITIS)
* **Masalah**: Membuat proyek dengan URL YouTube hanya menyisipkan baris database. Tidak ada video yang diunduh, tidak ada analisis yang berjalan, tidak ada klip yang dihasilkan. Pipeline benar-benar mati saat kedatangan.
* **Penyebab**: Handler `CreateProject` di `handlers/project.go` membuat catatan `Project` dan mengembalikan secara langsung. Handler `AddProjectVideo` (yang mendorong tugas Celery `process_video_ingest`) adalah endpoint terpisah yang tidak pernah dipanggil dari alur pembuatan proyek. `api.createProject()` frontend hanya mengenai `POST /projects`, tidak pernah `POST /projects/:id/videos`.
* **Solusi**: Menambahkan fungsi bantuan `ingestYouTubeVideo` yang berjalan secara asinkron melalui goroutine dari `CreateProject`. Fungsi ini mengekstrak ID video YouTube, mengambil metadata melalui `yt-dlp`, melakukan pemeriksaan kredit ABAC, membakar kredit, membuat baris `ProjectVideo`, memperbarui status proyek menjadi `ingesting`, dan mendorong tugas Celery `process_video_ingest`.

### 2. Analisis Tidak Pernah Memicu Rendering Klip (KRITIS)
* **Masalah**: Setelah `process_video_ingest` selesai, klip muncul di database dengan `export_status=NONE` dan tidak ada file MP4 yang dirender. Frontend menampilkan klip tetapi URL video selalu null.
* **Penyebab**: `process_video_ingest` adalah tugas monolitik yang menangani unduhan, transkripsi, analisis, dan persistensi kandidat — tetapi berhenti setelah menulis catatan database. Tugas `render_generated_short` ada dan sudah diimplementasikan sepenuhnya, tetapi tidak ada yang memanggilnya di seluruh kode sumber. Itu adalah kode mati.
* **Solusi**: Memodifikasi `process_video_ingest` untuk mengumpulkan ID GeneratedShort selama persistensi dan mengirimkan tugas Celery `render_generated_short` setelah analisis selesai.

### 3. Klip Yang Sudah Dirender Tidak Dapat Diakses Melalui API (SEDANG)
* **Masalah**: Bahkan setelah rendering, handler mengembalikan `video_url: null` untuk semua klip. Server file statis sudah dikonfigurasi tetapi pembangunan URL gagal.
* **Penyebab**: Handler membangun URL video sebagai `/storage/shorts/{ShortVideoID}.mp4`, membaca `ShortVideoID` dari model `GeneratedShort`. Tetapi `render_generated_short` di Python tidak pernah mengatur `short_video_id` — hanya mengatur `export_status` dan `export_on`. File disimpan sebagai `{gs.id}.mp4` tetapi ID tidak pernah ditulis kembali ke database.
* **Solusi**: Menambahkan penugasan `short_video_id` di `render_tasks.py` setelah rendering selesai.

### 4. Tidak Ada Pelayanan File Statis Untuk Klip Yang Sudah Dirender (SEDANG)
* **Masalah**: Handler mengembalikan URL seperti `/storage/shorts/{id}.mp4` tetapi tidak ada middleware file statis. Permintaan ke URL tersebut mengembalikan 404.
* **Penyebab**: Rute file statis tidak pernah ditambahkan ke aplikasi Fiber. Worker Python menulis klip ke `/app/exports/renders/` di dalam Docker, yang dipetakan ke `./processing-engine/exports/renders/` di host.
* **Solusi**: Menambahkan pelayanan file statis di `main.go`:
  ```go
  app.Static("/storage/shorts", "../processing-engine/exports/renders")
  ```

### 5. Ketidaksesuaian Pemetaan Status Ekspor Frontend (MINOR)
* **Masalah**: Klip yang selesai dirender ditampilkan sebagai "pending" di UI alih-alih "completed".
* **Penyebab**: Enum Python menggunakan `COMPLETED` dan `PROCESSING`, tetapi `mapShortStatus` frontend memetakan `DONE` → "completed" dan tidak memiliki entri untuk `COMPLETED` atau `PROCESSING`.
* **Solusi**: Memperbaiki pemetaan di `use-app-store.ts`.

### 6. Error Nesting JSX di Halaman Detail Proyek (MINOR)
* **Masalah**: Bagian Video dan Shorts muncul di luar pembungkus konten utama, merusak jarak dan tata letak.
* **Penyebab**: Tag `</div>` yang tersesat di baris 108 menutup pembungkus `space-y-6` terlalu cepat, mendorong semua konten berikutnya keluar darinya.
* **Solusi**: Menghapus `</div>` tambahan sehingga bagian Video dan Shorts benar-benar bersarang di dalam wadah `space-y-6`.

### 7. `projectName` Undefined di Halaman Editor (MINOR)
* **Masalah**: `ReferenceError: projectName is not defined` menjebak halaman editor.
* **Penyebab**: Variabel `projectId` dideklarasikan dari parameter rute, tetapi kode merujuk `projectName` (tidak pernah dideklarasikan) di 3 lokasi.
* **Solusi**: Mengganti semua referensi `projectName` dengan `project?.title || ''`.

## Fase Selesai / Fitur Terimplementasi

### Fase 1: Pemicu Otomatis Pencernaan dari Pembuatan Proyek
* **Detail Implementasi**: Menambahkan fungsi bantuan `ingestYouTubeVideo` di `handlers/project.go` yang berjalan sebagai goroutine dari `CreateProject`. Melakukan: ekstraksi ID YouTube → pengambilan metadata melalui yt-dlp → pemeriksaan kredit ABAC → pembakaran kredit → pembuatan baris ProjectVideo → pembaruan status menjadi `ingesting` → pengiriman tugas Celery.
* **Integritas Data**: URL YouTube dideteksi secara otomatis; proyek non-YouTube masih berfungsi (tidak ada pencernaan yang dipicu).
* **Optimasi**: Eksekusi asinkron berbasis goroutine sehingga respons API mengembalikan secara langsung.

### Fase 2: Pemicu Otomatis Rendering Setelah Analisis
* **Detail Implementasi**: Memodifikasi `process_video_ingest` di `tasks.py` untuk mengumpulkan ID GeneratedShort selama persistensi klip. Setelah mengatur status proyek menjadi `COMPLETED`, mengirimkan tugas Celery `render_generated_short` untuk setiap klip.
* **Integritas Data**: Setiap GeneratedShort mendapat tepat satu tugas rendering yang dikirimkan. Render yang gagal memperbarui `export_status=FAILED` tanpa menjebak pipeline.

### Fase 3: Tugas Render Mengatur Video ID
* **Detail Implementasi**: Memodifikasi `render_generated_short` di `render_tasks.py` untuk mengatur `short_video_id = str(gs.id)` setelah rendering berhasil. Ini menghubungkan file MP4 yang sudah dirender ke catatan database sehingga handler dapat membangun URL `/storage/shorts/{id}.mp4`.

### Fase 4: Pelayanan File Statis
* **Detail Implementasi**: Menambahkan `app.Static("/storage/shorts", "../processing-engine/exports/renders")` di `main.go`. Memetakan jalur volume Docker ke server file statis API Go.

### Fase 5: Status Polling Frontend & Banner Pemrosesan
* **Detail Implementasi**: Menambahkan polling interval 5 detik di halaman detail proyek (`project/[projectName]/page.tsx`) yang memanggil `api.getProjectStatus()` saat status adalah `ingesting/analyzing/rendering`. Menampilkan banner pemrosesan dengan spinner dan pesan deskriptif per status. Memuat ulang video secara otomatis dan menampilkan toast sukses saat status menjadi `completed`.
* **Integritas Data**: Polling berhenti pada unmount atau saat pemrosesan selesai/gagal.

### Fase 6: Perbaikan Pemetaan Status Frontend
* **Detail Implementasi**: Memperbaiki `mapShortStatus` di `use-app-store.ts` untuk mencocokkan nilai enum Python (`COMPLETED`, `PROCESSING`, `RENDERING`, `NONE`, `FAILED`).
* **Integritas Data**: Klip benar-benar mencerminkan status pemrosesan aktual mereka di UI.

## Arsitektur Pipeline (Setelah Perbaikan)

```
URL YouTube diajukan melalui frontend
    |
    v
[Go: Handler CreateProject]
    |  Mendeteksi URL YouTube → goroutine ingestYouTubeVideo()
    |  Mengambil metadata, membakar kredit, membuat ProjectVideo
    |  PushCeleryTask("process_video_ingest", [project_video_id])
    v
[Python: tugas process_video_ingest]
    |  Unduh video (yt-dlp) → Ekstrak audio (FFmpeg)
    |  Transkripsi (Whisper) → Deteksi adegan (PySceneDetect)
    |  Bangun kandidat (heatmap + skor viralitas)
    |  Rafinasi LLM kandidat teratas N (Gemini)
    |  Persistensi klip UNLOCKED + baris GeneratedShort (export_status=NONE)
    |  Persistensi klip LOCKED untuk Tier 4 unlock
    |  Atur status proyek = COMPLETED
    |  Kirim render_generated_short untuk setiap GeneratedShort
    v
[Python: tugas render_generated_short] (per klip)
    |  Unduh video sumber → Deteksi wajah untuk crop optimal
    |  Buat subtitle ASS (gaya karaoke)
    |  Potong potret FFmpeg (1080x1920) dengan subtitle burned
    |  Simpan MP4 ke /app/exports/renders/{id}.mp4
    |  Atur export_status=COMPLETED, short_video_id={id}
    v
[Go: Server file statis]
    |  Melayani /storage/shorts/{id}.mp4 dari exports/renders/
    v
[Frontend: Halaman detail proyek]
    |  Polling getProjectStatus() setiap 5 detik
    |  Saat completed → mengambil shorts → menampilkan klip dengan URL video
    |  Kartu klip menampilkan lencana status (pending/rendering/completed)
```

## Masalah yang Diketahui & Mitigasi

* **Redundansi Unduh Render**: Setiap tugas `render_generated_short` mengunduh video sumber secara independen melalui yt-dlp. Untuk proyek dengan 10 klip, video yang sama diunduh 10 kali. **Mitigasi**: Pertimbangkan caching video yang diunduh di direktori bersama, atau unduh sekali dan teruskan jalur ke tugas render.
* **Tanpa Pembuatan Thumbnail**: Tugas `render_generated_short` menghasilkan file MP4 tetapi tidak membuat gambar thumbnail. `ShortThumbnailURL` selalu null. **Mitigasi**: Menambahkan ekstraksi thumbnail ffmpeg ke tugas render (misalnya, ekstrak frame di titik tengah).
* **Pemrosesan Unggahan File**: Unggahan file (`POST /projects/upload`) membuat catatan proyek tetapi tidak memicu pipeline pemrosesan — hanya URL YouTube yang memicu pencernaan. **Mitigasi**: Menghubungkan jalur unggah ke antrean pemrosesan Celery.
* **Direktori `uploads/`**: File disimpan ke `uploads/` lokal di direktori kerja backend Go. Tidak siap produksi. **Mitigasi**: Migrasi ke penyimpanan objek (S3/R2).
* **Biner Terkompilasi di Repo**: Biner `go-backend/api` dilacak di git. **Mitigasi**: Ditambahkan ke `.gitignore`.
