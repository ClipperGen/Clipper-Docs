# Clipper-Gen Pipeline Wiring Report: End-to-End Clip Generation

## Executive Summary / Summary of Work
The Clipper-Gen platform's core feature — automatically generating portrait clips from YouTube videos — was broken at multiple points in the pipeline. The Go backend created projects but never triggered video ingestion. The Python processing engine analyzed videos and generated clip recommendations but never triggered rendering. And the frontend had no mechanism to display processing progress or fetch rendered clips. This session traced the full pipeline end-to-end, identified every broken link, and wired them together. The platform now completes the full flow: YouTube URL submission, automatic video download, transcription, AI analysis, clip candidate generation, portrait rendering with burned subtitles, and frontend display with live status polling.

## Bug Details & Resolusi

### 1. Project Creation Never Triggers Video Ingestion (CRITICAL)
* **The Problem (Masalah)**: Creating a project with a YouTube URL only inserted a database row. No video was downloaded, no analysis ran, no clips were generated. The pipeline was completely dead on arrival.
* **The Cause (Penyebab)**: `CreateProject` handler in `handlers/project.go` created a `Project` record and returned immediately. The `AddProjectVideo` handler (which pushes the `process_video_ingest` Celery task) was a separate endpoint that was never called from the project creation flow. The frontend's `api.createProject()` only hit `POST /projects`, never `POST /projects/:id/videos`.
* **The Solving (Solusi)**: Added `ingestYouTubeVideo` helper function that runs asynchronously via goroutine from `CreateProject`. It extracts the YouTube video ID, fetches metadata via `yt-dlp`, performs ABAC credit checks, burns credits, creates a `ProjectVideo` row, updates project status to `ingesting`, and pushes the `process_video_ingest` Celery task. The YouTube URL is auto-detected in `CreateProject`:
  ```go
  if sourceURL != "" {
      if videoID, err := utils.ExtractYouTubeVideoID(sourceURL); err == nil && videoID != "" {
          go ingestYouTubeVideo(user, project, sourceURL)
      }
  }
  ```

### 2. Analysis Never Triggers Clip Rendering (CRITICAL)
* **The Problem (Masalah)**: After `process_video_ingest` completed, clips appeared in the database with `export_status=NONE` and no rendered MP4 files. The frontend showed clips but video URLs were always null.
* **The Cause (Penyebab)**: `process_video_ingest` was a monolithic task that handled download, transcription, analysis, and candidate persistence — but it stopped after writing database records. The `render_generated_short` task existed and was fully implemented, but nothing in the codebase ever called it. It was dead code.
* **The Solving (Solusi)**: Modified `process_video_ingest` to collect GeneratedShort IDs during persistence and dispatch `render_generated_short` Celery tasks after analysis completes:
  ```python
  generated_short_ids: list[str] = []
  for idx, cand in enumerate(top, start=1):
      # ... analysis and persistence ...
      gs_id = await _persist_unlocked_clip(...)
      if gs_id:
          generated_short_ids.append(gs_id)

  # After setting project status to COMPLETED
  for gs_id in generated_short_ids:
      celery_app.send_task("render_generated_short", args=[gs_id])
  ```
  Also changed `_persist_unlocked_clip` return type from `None` to `str` to return the GeneratedShort ID.

### 3. Rendered Clips Not Accessible via API (MODERATE)
* **The Problem (Masalah)**: Even after rendering, the Go handler returned `video_url: null` for all clips. The static file server was configured but the URL construction failed.
* **The Cause (Penyebab)**: The Go handler constructs video URLs as `/storage/shorts/{ShortVideoID}.mp4`, reading `ShortVideoID` from the `GeneratedShort` model. But `render_generated_short` in Python never set `short_video_id` — it only set `export_status` and `export_on`. The file was saved as `{gs.id}.mp4` but the ID was never written back to the database.
* **The Solving (Solusi)**: Added `short_video_id` assignment in `render_tasks.py` after rendering completes:
  ```python
  gs_done.short_video_id = str(gs.id)
  await gs_done.save(update_fields=["export_status", "export_on", "short_video_id"])
  ```
  Also verified `app.Static("/storage/shorts", "../processing-engine/exports/renders")` in `main.go` serves the files correctly.

### 4. No Static File Serving for Rendered Clips (MODERATE)
* **The Problem (Masalah)**: The Go handler returned URLs like `/storage/shorts/{id}.mp4` but no static file middleware existed. Requests to those URLs returned 404.
* **The Cause (Penyebab)**: The static file route was never added to the Fiber app. The Python worker writes clips to `/app/exports/renders/` inside Docker, which maps to `./processing-engine/exports/renders/` on the host.
* **The Solving (Solusi)**: Added static file serving in `main.go`:
  ```go
  app.Static("/storage/shorts", "../processing-engine/exports/renders")
  ```

### 5. Frontend Export Status Mapping Mismatch (MINOR)
* **The Problem (Masalah)**: Clips that finished rendering showed as "pending" in the UI instead of "completed".
* **The Cause (Penyebab)**: The Python enum uses `COMPLETED` and `PROCESSING`, but the frontend's `mapShortStatus` mapped `DONE` → "completed" and had no entry for `COMPLETED` or `PROCESSING`.
* **The Solving (Solusi)**: Fixed the mapping in `use-app-store.ts`:
  ```typescript
  const map: Record<string, GeneratedShort["status"]> = {
    NONE: "pending",
    PENDING: "pending",
    PROCESSING: "rendering",
    RENDERING: "rendering",
    COMPLETED: "completed",
    FAILED: "failed",
  };
  ```

### 6. JSX Nesting Error in Project Detail Page (MINOR)
* **The Problem (Masalah)**: Videos and Shorts sections appeared outside the main content wrapper, breaking spacing and layout.
* **The Cause (Penyebab)**: A stray `</div>` at line 108 prematurely closed the `space-y-6` wrapper div, pushing all subsequent content outside it.
* **The Solving (Solusi)**: Removed the extra `</div>` so Videos and Shorts sections are properly nested inside the `space-y-6` container.

### 7. `projectName` Undefined in Editor Page (MINOR)
* **The Problem (Masalah)**: `ReferenceError: projectName is not defined` crashed the editor page.
* **The Cause (Penyebab)**: The variable `projectId` was declared from route params, but the code referenced `projectName` (never declared) in 3 locations.
* **The Solving (Solusi)**: Replaced all `projectName` references with `project?.title || ''`.

## Pipeline Architecture (After Fix)

```
YouTube URL submitted via frontend
    |
    v
[Go: CreateProject handler]
    |  Detects YouTube URL → goroutine ingestYouTubeVideo()
    |  Fetches metadata, burns credits, creates ProjectVideo
    |  PushCeleryTask("process_video_ingest", [project_video_id])
    v
[Python: process_video_ingest task]
    |  Download video (yt-dlp) → Extract audio (FFmpeg)
    |  Transcribe (Whisper) → Scene detect (PySceneDetect)
    |  Build candidates (heatmap + virality scoring)
    |  LLM refine top N candidates (Gemini)
    |  Persist UNLOCKED clips + GeneratedShort rows (export_status=NONE)
    |  Persist LOCKED clips for Tier 4 unlock
    |  Set project status = COMPLETED
    |  Dispatch render_generated_short for each GeneratedShort
    v
[Python: render_generated_short task] (per clip)
    |  Download source video → Face detection for optimal crop
    |  Generate ASS subtitles (karaoke style)
    |  FFmpeg portrait cut (1080x1920) with burned subtitles
    |  Save MP4 to /app/exports/renders/{id}.mp4
    |  Set export_status=COMPLETED, short_video_id={id}
    v
[Go: Static file server]
    |  Serves /storage/shorts/{id}.mp4 from exports/renders/
    v
[Frontend: Project detail page]
    |  Polls getProjectStatus() every 5 seconds
    |  On completed → fetches shorts → displays clips with video URLs
    |  Clip cards show status badges (pending/rendering/completed)
```

## Known Issues & Mitigasi

* **Render Download Redundancy**: Each `render_generated_short` task downloads the source video independently via yt-dlp. For a project with 10 clips, the same video is downloaded 10 times. **Mitigation**: Consider caching the downloaded video in a shared temp directory, or downloading once and passing the path to render tasks.
* **No Thumbnail Generation**: The `render_generated_short` task generates MP4 files but does not create thumbnail images. `ShortThumbnailURL` is always null. **Mitigation**: Add ffmpeg thumbnail extraction to the render task (e.g., extract frame at midpoint).
* **File Upload Processing**: File uploads (`POST /projects/upload`) create project records but do not trigger the processing pipeline — only YouTube URLs trigger ingestion. **Mitigation**: Connect upload path to Celery processing queue.
* **`uploads/` Directory**: Files saved to local `uploads/` in Go backend working directory. Not production-ready. **Mitigation**: Migrate to object storage (S3/R2).
* **Compiled Binary in Repo**: `go-backend/api` binary tracked in git. **Mitigation**: Add to `.gitignore`.
