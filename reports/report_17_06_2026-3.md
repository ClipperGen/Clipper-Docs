# Laporan Admin Panel: 5 Fitur Backend + Frontend

## Ringkasan Eksekutif
Sesi ini mengimplementasikan lima fitur sekaligus pada Admin Panel: penggantian placeholder Activity Log dengan data audit real-time, pencarian server-side pada daftar pengguna, ekspor CSV, aksi bulk (massal) untuk aktivasi/membrukan pengguna, serta logging audit yang lengkap untuk setiap perubahan admin. Semua perubahan meliputi backend Go dan frontend Admin FE.

## Fitur 1: Dashboard — Activity Log Real-time dari Audit Logs

### Masalah
Dashboard menampilkan lima entri activity log yang di-hardcode. Tidak ada indikator aktivitas admin yang sebenarnya.

### Solusi
* Menambahkan `GET /admin/audit-logs` (handler `AdminListAuditLogs`) — mengembalikan hingga 200 entri audit log terbaru, diurutkan per waktu.
* Menambahkan hook `useAuditLogs(limit)` ke Admin FE.
* Mengganti seluruh section "System Activity Log" pada `page.tsx` dengan data audit log real-time.
* Setiap entri menampilkan: icon berdasarkan jenis aksi (role update, tier update, config update, bulk status, grant tokens), deskripsi detail, timestamp relatif, dan target ID (dipotong).
* Section "Predictive Analytics Engine" placeholder dihapus — dashboard sekarang hanya menampilkan stat cards + audit log.

### File Diubah
* `Clipper_BE/.../handlers/admin.go` — tambah `AdminListAuditLogs`
* `Clipper_BE/.../cmd/api/main.go` — register `GET /admin/audit-logs`

* `Clipper_Admin_FE/src/hooks/use-api.ts` — tambah `useAuditLogs()`

## Fitur 2: Pencarian Server-Side

### Masalah
Pencarian dilakukan di sisi klien dengan `Array.filter()`. Untuk banyak pengguna, seluruh array dimuat ke browser.

### Solusi
* Backend `AdminListUsers` menerima parameter query `?search=`. Query menggunakan `ILIKE` terhadap `email` dan `id::text`.
* Hook `useUsers()` diubah untuk menerima `search?: string`, dimasukkan ke `queryKey` React Query.
* Input pencarian frontend menggunakan debounce 350ms — API hanya dipanggil setelah user berhenti mengetik.
* Saat kolom kosong, seluruh data dimuat tanpa filter.

### File Diubah
* `Clipper_BE/.../handlers/admin.go` — `AdminListUsers` tambah `c.Query("search")`
* `Clipper_Admin_FE/src/hooks/use-api.ts` — `useUsers(search?)`

## Fitur 4: Bulk Actions (Aktivasi/Membrukan Massal)

### Solusi
* Kolom checkbox ditambahkan ke tabel pengguna — header checkbox memilih/membatalkan semua, checkbox per-baris memilih per pengguna.
* State `selectedIds: Set<string>` melacak ID yang dipilih.
* Bilah aksi bulk muncul di atas tabel saat ada yang dipilih — menampilkan jumlah yang dipilih, tombol "Activate" (hijau), tombol "Revoke" (merah), dan tombol tutup.
* Backend `POST /admin/users/bulk-status` menerima `{ user_ids: string[], is_active: boolean }` — melakukan UPDATE massal (`WHERE id IN (?)`) dengan batas 100 pengguna per permintaan.
* Endpoint mencatat audit log `BULK_UPDATE_STATUS` dengan jumlah baris yang terpengaruh.
* Setelah berhasil, daftar pengguna di-refresh otomatis.

## Fitur 5: Log Audit Lengkap

### Masalah
Hanya `AdminUpdateUserRole`, `AdminUpdateConfig`, dan `AdminUpdateTier` yang mencatat audit log. `AdminGrantTokens` dan `AdminUpdateUserStatus` tidak mencatat apa pun.

### Solusi

## Masalah yang Diketahui & Mitigasi

* **Debounce Search**: Menggunakan timer global (`globalThis.__searchTimer`). Aman untuk mode Strict React karena timer di-reset.
* **CSV Export**: Menggunakan `Blob` + `<a>` click. Aman di semua browser modern.
* **Bulk Action Limit**: Dibatasi 100 pengguna per permintaan untuk mencegah query besar.
* **Audit Log Tanpa Admin Name**: Log hanya menyimpan `admin_id`, bukan nama. Nama bisa ditambahkan via JOIN jika diperlukan.
