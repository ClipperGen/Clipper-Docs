# Laporan Rebuild Admin Frontend: API Client, React Query, dan Tipe Terbagi

## Ringkasan Eksekutif
Admin frontend (`Clipper_Admin_FE`) sebelumnya berjalan tanpa integrasi API — setiap halaman menggunakan data hardcode, tidak ada tipe bersama, tidak ada error handling, dan port default salah (`8005` alih-alih `8000`). Sesi ini melakukan rebuild penuh: membuat klien API terbagi dengan autentikasi Clerk, hook React Query untuk setiap endpoint admin, tipe TypeScript yang dibagikan, dan menulis ulang semua 5 halaman untuk mengonsumsi data API nyata.

---

## Status Proyek & Metadata
* **Progres**: Admin FE sepenuhnya terhubung ke backend Go; data hardcode dihapus, semua halaman menggunakan hook API
* **Arsitektur**: Monorepo — Go Fiber API + Python Celery worker + Next.js 16 user FE + Next.js 16 admin FE
* **Tumpukan Frontend**: Next.js 16 + React 19 + TanStack React Query + Clerk Auth + Tailwind CSS
* **Tumpukan Backend**: Go Fiber v2 + PostgreSQL + Redis + Qdrant
* **Branch**: `wired` (semua repo)
* **Status Build**: TypeScript — 0 error; Docker — semua image berhasil dibangun

---

## Detail Bug & Resolusi

### 1. Port API Default Salah
* **Masalah**: Admin FE default API URL mengarah ke `localhost:8005`, tetapi Go backend berjalan di `localhost:8000`. Semua permintaan API akan gagal.
* **Penyebab**: Kesalahan konfigurasi awal — port tidak diverifikasi terhadap `docker-compose.yml` dan `main.go`.
* **Solusi**: Membuat `.env.local` dengan `NEXT_PUBLIC_API_URL=http://localhost:8000/api/v1`. Menghapus hardcoded fallback di `api-client.ts`.

### 2. Tidak Ada Dependensi `@tanstack/react-query`
* **Masalah**: Build TypeScript gagal — `Cannot find module '@tanstack/react-query'`.
* **Penyebab**: Hook React Query ditulis tanpa menambahkan dependensi ke `package.json`.
* **Solusi**: Menjalankan `bun add @tanstack/react-query`. Build bersih setelah instalasi.

## Fase Selesai / Fitur Terimplementasi

### Fase 1: Tipe TypeScript Terbagi
* **Detail Implementasi**: Membuat `src/types/index.ts` dengan 4 antarmuka: `AdminUser` (17 field), `AdminTier` (8 field), `SystemConfig` (5 field), `HealthStats` (10 field). Semua tipe diekspor dan digunakan oleh klien API, hook, dan komponen.
* **Integritas Data**: Tipe mencocokkan JSON response dari Go backend (tag `json:"..."` pada struct Go). Tidak ada `any` yang lolos dari tipe.

### Fase 2: Klien API Terbagi dengan Autentikasi Clerk
* **Detail Implementasi**: Membuat `src/lib/api-client.ts` dengan hook `useAdminApi()` yang mengembalikan objek `{ get, post, patch, put }`. Setiap metode secara otomatis menyertakan header autentikasi Clerk (`Bearer` token) melalui `useAuth().getToken()`. Base URL dibaca dari `NEXT_PUBLIC_API_URL`.
* **Integritas Data**: Token Clerk diperbarui secara otomatis — tidak ada token hardcode. Error HTTP dilempar sebagai exception untuk ditangani oleh React Query.

### Fase 3: Hook React Query untuk Semua Endpoint Admin
* **Detail Implementasi**: Membuat `src/hooks/use-api.ts` dengan 9 hook:
  - `useHealth()` — `GET /admin/system/health`
  - `useUsers(params?)` — `GET /admin/users`
  - `useGrantTokens()` — `POST /admin/users/:id/grant-tokens`
  - `useUpdateUserRole()` — `PATCH /admin/users/:id/role`
  - `useUpdateUserStatus()` — `PATCH /admin/users/:id/status`
  - `useSubscriptions()` — `GET /admin/subscriptions`
  - `useUpdateSubscription()` — `PUT /admin/subscriptions/:id`
  - `useConfigs()` — `GET /admin/configs`
  - `useUpdateConfig()` — `PATCH /admin/configs/:key`
* **Integritas Data**: Setiap mutation menggunakan `invalidateQueries` untuk menyegarkan data terkait secara otomatis. Parameter query bersifat opsional dengan default yang masuk akal.

### Fase 4: Provider React Query
* **Detail Implementasi**: Membuat `src/app/providers.tsx` yang membungkus aplikasi dengan `QueryClientProvider`. Mengkonfigurasi `QueryClient` dengan default staleTime 30 detik dan retry 1. Menyuntikkan provider ke `layout.tsx`.
* **Integritas Data**: Query client dibuat sekali per aplikasi (bukan per render) untuk mencegah kebocoran state.

### Fase 5: Penulisan Ulang 5 Halaman

#### 5a. Dashboard (`page.tsx`)
* **Detail Implementasi**: Menghapus 6 placeholder stat card hardcode. Menggunakan `useHealth()` untuk menampilkan metrik nyata: total pengguna, total proyek, total klip, total transaksi, status worker, status Redis. Menambahkan state pemuatan (grid skeleton) dan error.
* **Data Mock Dihapus**: 6 stat card, 5 activity log entries, 3 pipeline bars, 2 status cards.

#### 5b. Manajemen Pengguna (`users/page.tsx`)
* **Detail Implementasi**: Menghapus data `mockUsers` (5 item). Menggunakan `useUsers()` untuk menampilkan daftar pengguna nyata dengan pencarian, filter status, dan aksi: grant token, ubah peran, ubah status. Menambahkan dialog konfirmasi untuk setiap aksi.
* **Data Mock Dihapus**: 5 pengguna hardcode dengan ID, nama, email, peran, status.

#### 5c. Manajemen Langganan (`subscriptions/page.tsx`)
* **Detail Implementasi**: Menghapus data `mockTiers` (3 tier). Menggunakan `useSubscriptions()` untuk menampilkan tier langganan nyata dengan edit inline dan grafik distribusi pengguna. Menambahkan form edit untuk harga, batas token, dan batas proyek.
* **Data Mock Dihapus**: 3 tier hardcode dengan nama, harga, fitur, batas.

#### 5d. Kesehatan Sistem (`health/page.tsx`)
* **Detail Implementasi**: Menghapus status hardcode (PostgreSQL, Redis, Worker, API). Menggunakan `useHealth()` untuk menampilkan status nyata dari endpoint kesehatan. Menambahkan indikator hijau/merah untuk setiap komponen.
* **Data Mock Dihapus**: 4 status komponen hardcode.

#### 5e. Pengaturan (`settings/page.tsx`)
* **Detail Implementasi**: Menggunakan `useConfigs()` untuk menampilkan konfigurasi sistem nyata. Menambahkan form edit untuk setiap config key dengan mutation `useUpdateConfig()`. Menambahkan toast notifikasi untuk sukses/gagal.
* **Data Mock Dihapus**: Pengaturan hardcode untuk maintenance mode, auto-publish, render quality.

## Masalah yang Diketahui & Mitigasi

* **Data Placeholder Tersisa**: Activity Log (5 entri hardcode), Pipeline Saturation (bar visual), PostgreSQL/Worker status (kartu hardcode) masih ada di dashboard. **Mitigasi**: Bagian ini akan dihapus atau dihubungkan saat endpoint analitik tersedia.
* **Tanpa Pencarian Server-Side**: Pencarian pengguna dilakukan di sisi klien (filter array). **Mitigasi**: Tambahkan parameter `search` ke `GET /admin/users` saat jumlah pengguna meningkat.
* **Autentikasi Dev Mode**: Saat ini berjalan dengan Clerk dev bypass. **Mitigasi**: Verifikasi alur auth Clerk dengan token nyata sebelum produksi.
