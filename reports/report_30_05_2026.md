# Clipper-Gen Frontend Modernization Report: Full API Integration & Bug Fixes

## Executive Summary / Summary of Work
The Clipper-Gen frontend (Next.js 16 + Clerk Auth) was previously operating with broken data flows — mock data, type mismatches against the Go backend, and several runtime errors preventing any real functionality. This session performed a full-stack wiring pass: aligning all frontend API interfaces to the Go backend's actual response shapes, removing hardcoded mock data from 6+ components, adding a real file upload endpoint to the backend, and fixing two critical runtime errors (infinite re-render loop and undefined function reference). The frontend now loads real user data, projects, activities, transactions, and notifications from the API, and file uploads are persisted to disk via a new multipart endpoint.

## Bug Details & Resolusi

### 1. "Maximum update depth exceeded" — Infinite Re-render Loop (sidebar.tsx:215)
* **The Problem (Masalah)**: React runtime error — `Maximum update depth exceeded` triggered in `SidebarContent` component at the achievements tooltip. The entire dashboard was unusable due to infinite re-renders.
* **The Cause (Penyebab)**: `useApi()` hook (`frontend/src/lib/api.ts:270`) created `new ClipperApi(getToken)` on **every render**, producing a new object reference each time. This was passed into `useDashboardData()` which had `useEffect(() => { fetchAll(api) }, [api, fetchAll])` — the changing `api` reference triggered the effect on every render cycle, which updated the store, which re-rendered the sidebar, which recreated `api`, ad infinitum.
* **The Solving (Solusi)**: Memoized the `ClipperApi` instance with `useMemo`:
  ```typescript
  // Before (broken)
  export function useApi(): ClipperApi {
    const { getToken } = useAuth();
    return new ClipperApi(getToken);
  }

  // After (fixed)
  export function useApi(): ClipperApi {
    const { getToken } = useAuth();
    return useMemo(() => new ClipperApi(getToken), [getToken]);
  }
  ```

### 2. "addProject is not defined" — Runtime ReferenceError (project/page.tsx, editor/page.tsx)
* **The Problem (Masalah)**: `ReferenceError: addProject is not defined` crashed both the Project page and Editor placeholder page when rendering `FileUploadDialog`.
* **The Cause (Penyebab)**: `simulateUpload()` in `file-upload-dialog.tsx:209` called `addProject({...})` — a function that was never imported or defined in the component's scope. This was leftover from a removed mock pattern where `addProject` was supposed to come from the store but was replaced by `createProject`.
* **The Solving (Solusi)**: Replaced the entire `simulateUpload` mock with a real `handleUpload` function that calls `api.uploadProject(file)` for each file, then refreshes the project list via `fetchProjects(api)`. Also fixed `handleUrlImport` to use `api.createProject()` directly instead of the store wrapper.

### 3. "socket hang up" / Frontend 500 errors behind Cloudflare tunnel
* **The Problem (Masalah)**: Next.js 16 middleware behind Cloudflare tunnel produced `socket hang up` errors due to deprecated `x-middleware-rewrite` header pointing to `localhost`.
* **The Cause (Penyebab)**: Next.js 16 deprecated the `middleware.ts` convention in favor of "proxy" mode. The middleware was rewriting to `localhost`, which broke when accessed through the Cloudflare tunnel.
* **The Solving (Solusi)**: Applied the admin's workaround — middleware strips `x-middleware-rewrite` headers pointing to `localhost` before forwarding the response:
  ```typescript
  const rewrite = res.headers.get("x-middleware-rewrite");
  if (rewrite && rewrite.includes("localhost")) {
    const nextRes = NextResponse.next();
    res.headers.forEach((v, k) => {
      if (k.toLowerCase() !== "x-middleware-rewrite") {
        nextRes.headers.set(k, v);
      }
    });
    return nextRes;
  }
  ```

### 4. Data Never Loads — Chicken-and-Egg Auth Gate
* **The Problem (Masalah)**: Dashboard showed empty state permanently — no data ever loaded from the API.
* **The Cause (Penyebab)**: `useDashboardData()` gated all fetch calls on `isLoggedIn`, which was `false` by default and never set by the Clerk auth flow. The hook never called `fetchAll(api)`.
* **The Solving (Solusi)**: Removed the `isLoggedIn` gate. With Clerk, auth is handled by middleware — the hook calls `fetchAll(api)` directly. If the token is invalid, API calls fail gracefully with 401.

## Known Issues & Mitigasi
* **Video Processing Pipeline**: File uploads are saved to disk and project records created, but the actual video processing (FFmpeg/Whisper/Celery) is not yet triggered for uploaded files — only YouTube URL ingestion triggers the full pipeline. **Mitigation**: Integrate file upload path with the Celery processing queue.
* **Frontend Submodule Detached HEAD**: The frontend submodule commit was made in detached HEAD state. **Mitigation**: Pushed successfully to `master` via `HEAD:refs/heads/master`, but should be on a proper branch for future work.
* **`uploads/` Directory**: Files are saved to a local `uploads/` directory in the Go backend working directory. Not suitable for production — needs object storage (S3/R2). **Mitigation**: Track for infrastructure migration.
* **Compiled Binary in Repo**: `go-backend/api` (compiled ELF binary) shows as modified in git. Should be added to `.gitignore`.
